标准命令：

**首次尝试失败**：报错链路是 `bfcl-eval → google-genai → google-auth → cryptography`，uv试图从源码构建 `cryptography` 的Rust扩展（`cryptography-rust`），触发 `cargo`/`maturin` 去连 `crates-io`，进而卡在已失效的 `http://127.0.0.1:10090` 代理上。

**根因**：`cryptography` 从 **49.0.0 版本开始不再发布 macOS x86_64 (Intel) 的预编译wheel**（只保留 `macosx_11_0_arm64`），本机虽是Apple Silicon硬件，但项目venv的Python是x86_64架构（经Rosetta运行），uv解析到最新版49.0.0后找不到匹配wheel，被迫退回源码构建。46.0.x～48.0.x仍发布 `macosx_10_9_universal2` wheel（同时含x86_64和arm64切片），可以直接下载。

**解决**：加约束 `cryptography<49`，改为：

```bash
cd /Users/shenweiming/Projects/论文/experiments
GIT_CONFIG_GLOBAL=/dev/null uv pip install --python .venv/bin/python \
  -e ./gorilla/berkeley-function-call-leaderboard "cryptography<49"
```

安装成功后发现另一个独立问题：`bfcl_eval` 直接依赖 `qwen-agent`（用于Qwen系列模型的handler），而 `qwen-agent` 自身又依赖 `soundfile`，但uv默认解析未把它拉进来，导致 `import bfcl_eval` 时因 `model_config.py` 顶部一次性 import 所有handler（包括Qwen的）而报 `ModuleNotFoundError: No module named 'soundfile'`。补装即可：

```bash
GIT_CONFIG_GLOBAL=/dev/null uv pip install --python .venv/bin/python soundfile
```

### 2.3 BFCL接入DeepSeek的机制（调研结论）

BFCL对DeepSeek的官方支持是**专用handler**，而不是走通用的"Pre-existing OpenAI-compatible Endpoint"机制：

- 专用handler：`bfcl_eval/model_handler/api_inference/deepseek.py` 中的 `DeepSeekAPIHandler`，继承自 `OpenAICompletionsHandler`，内部直接用官方 `openai` SDK 构造client：
  ```python
  self.client = OpenAI(base_url="https://api.deepseek.com", api_key=os.getenv("DEEPSEEK_API_KEY"))
  ```
  即：只要设置环境变量 `DEEPSEEK_API_KEY`（`experiments/.env` 里已有，未改动），该handler就能工作。
- 模型名到API调用名的映射在 `bfcl_eval/constants/model_config.py` 的 `api_inference_model_map` 字典里，例如官方自带的：
  - `"DeepSeek-V3.2-Exp-FC"` → `model_name="deepseek-chat"`，`is_fc_model=True`（走Function Calling模式）
  - `"DeepSeek-V3.2-Exp-thinking"` → `model_name="deepseek-reasoner"`
- 注意：README中提到的 `REMOTE_OPENAI_BASE_URL` / `REMOTE_OPENAI_API_KEY` / `REMOTE_OPENAI_TOKENIZER_PATH` 环境变量是给 `local_inference/base_oss_handler.py`（自建vLLM/SGLang服务）用的**通用OpenAI兼容endpoint机制**，DeepSeek官方handler不读取这几个变量，二者是两条独立路径。

**因为本项目实际可用的模型是 `deepseek-v4-flash` / `deepseek-v4-pro`（不是BFCL自带列表里的 `deepseek-chat`/`deepseek-reasoner`）**，已在源码里新增两个模型条目复用现成的 `DeepSeekAPIHandler`（编辑的是仓库里的文件，属于本地补丁，不是上游更新）：

- 文件：`experiments/gorilla/berkeley-function-call-leaderboard/bfcl_eval/constants/model_config.py`（在 `api_inference_model_map` 中新增）
- 文件：`experiments/gorilla/berkeley-function-call-leaderboard/bfcl_eval/constants/supported_models.py`（同步加入清单，仅影响可读性/`bfcl models`列表）

说明：
- "平均Agent成本/User成本" 直接取自tau2 `results.json` 里每条simulation的 `agent_cost` / `user_cost` 字段（美元）。
- "平均Agent Prompt/Completion Tokens" 是遍历每条simulation的 `messages` 数组中 `role == "assistant"` 的消息，累加其 `usage.prompt_tokens` / `usage.completion_tokens` 字段（agent自己每一步调用的token数总和），再对有效任务取平均；不含user模拟器一侧的token消耗。
- "平均消息数" 是每条simulation的 `messages` 列表长度（含assistant/user/tool三种角色）的平均值。
- Pass^1 = reward达到1.0的任务占（有效任务数）的比例。
- deepseek-v4-pro 单任务成本约为flash的2.5倍左右，符合pro模型更贵的预期；同时pro在retail域的完成质量(reward)反而更高，flash在token消耗上通常更高（可能因为flash的回复/推理token更冗长）。
